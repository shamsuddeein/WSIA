WSIA

Web Security Intelligence Architecture

Build Plan & Execution Guide



Overview

This is the build order for WSIA — a security intelligence pipeline that scrapes, cleans, stores, and serves vulnerability data. The goal is a system where data integrity comes first and features follow after the foundation is stable.

The document covers 8 phases. Each phase has a clear finish condition, a troubleshooting table, and notes on what can go wrong. Don't start the next phase until the current one passes its checks.



System Flow

The full pipeline, once complete:



1

Scraper  —  one source to start, rekt.news

2

Analytics  —  clean and tag the raw data

3

Reports DB  —  structured, deduplicated storage

4

API  —  serve data via REST endpoints

5

Search  —  filter by category, severity, keyword

6

AI assistant  —  optional layer, added last



Build in this order. The AI layer is last because it depends on everything before it working correctly. If the data is messy, AI summaries will be wrong. There's no shortcut around that.

⚠  Note:  The hash field is the deduplication key. Before inserting any record, check if a report with that hash already exists. Set unique=True and db_index=True at the database level — don't rely on application-layer checks alone. A race condition during automated scraping can still insert duplicates without the database constraint.



Why source and raw_data matter

source lets you filter by provider immediately — once Phase 5 adds a second scraper, you'll want to query 'all rekt.news reports' without touching the URL. Add it now, not later.

raw_data is a JSONField that holds whatever the source gave you that didn't map cleanly to your schema. Different scrapers return different shapes. Without this field, you'd either lose that data or change your schema every time you add a source. Store it raw, ignore it until you need it.



Database indexes

Add these to HackReport.Meta before the first migration. Indexes are cheap to add early and expensive to add later on a table with millions of rows.

class HackReport(models.Model):

    # ... fields ...



    class Meta:

        indexes = [

            models.Index(fields=['title']),

            models.Index(fields=['severity']),

            models.Index(fields=['created_at']),

            models.Index(fields=['published_at']),

            models.Index(fields=['source']),

            # hash is already indexed via unique=True

        ]

Why a services/ layer

Keep models thin. Models should define data structure and relationships — nothing else. Business logic (is this a duplicate? which category fits this text?) lives in services/. This makes each piece testable in isolation and keeps models from growing into unmaintainable blobs over time.

# reports/services/dedup_service.py

import hashlib

from reports.models import HackReport



def compute_hash(url: str) -> str:

    return hashlib.sha256(url.encode()).hexdigest()



def is_duplicate(url: str) -> bool:

    return HackReport.objects.filter(hash=compute_hash(url)).exists()

Rate limiting and timeouts

⚠  Note:  Always set a timeout on requests.get(). Without one, a hung connection will freeze your scraper indefinitely.

Add a delay between requests: time.sleep(2). Hitting a site without delays can get your IP blocked.

If you get a 429 (Too Many Requests), back off and retry after 60 seconds minimum.

If you get a 403, the site may be blocking scrapers. Check their robots.txt before going further.



What not to store

✗  Avoid:  Full article HTML — store only the fields you defined in the model.

Author names or personal details from the articles.

Copyrighted article body text in full — a short description or excerpt is fine.

Raw cookies, session tokens, or any auth data from the response headers.



Hard limits for this phase

✗  No second website.

✗  No AI integration.

✗  No data cleaning yet — raw data is fine.

Phase 4  Data Cleaning & Normalization



Turn raw scraped text into structured data the API can serve usefully. Rule-based logic only — no ML, no AI.



Build the analytics app

Create analytics app: python manage.py startapp analytics

clean_text(text)  —  strip HTML tags, collapse whitespace, remove non-UTF-8 characters

normalize_report(report)  —  apply clean_text to title and description, set is_processed=True

assign_category(report)  —  keyword matching: 'reentrancy' → Reentrancy, 'oracle' → Oracle, etc.

assign_severity(report)  —  rule-based: dollar amounts > $10M → high, mentions of 'drained' → critical



Keyword rules (starting set)

CATEGORY_RULES = {

    'reentrancy':  'Reentrancy',

    'oracle':      'Oracle Manipulation',

    'flash loan':  'Flash Loan',

    'bridge':      'Bridge Exploit',

    'private key': 'Key Compromise',

}



def assign_category(text):

    text_lower = text.lower()

    for keyword, category_name in CATEGORY_RULES.items():

        if keyword in text_lower:

            cat, _ = Category.objects.get_or_create(name=category_name)

            return cat

    return None

Pipeline flow

@shared_task

def run_pipeline():

    reports = scrape_rekt()          # Phase 3

    for report in reports:

        if not is_duplicate(report):  # hash check

            raw = store_raw(report)   # save is_processed=False

            normalize_report(raw)     # Phase 4

            assign_category(raw)      # Phase 4



Rollback plan

⚠  Note:  If the pipeline corrupts data, restore from backup and delete all records where is_processed=False before re-running.

Keep a created_at timestamp on every record — it's your audit trail for when records entered the system.

Search endpoint

# api/views.py

from django.db.models import Q



class ReportSearchView(generics.ListAPIView):

    serializer_class = HackReportSerializer



    def get_queryset(self):

        q = self.request.query_params.get('q', '')

        return HackReport.objects.filter(

            Q(title__icontains=q) | Q(description__icontains=q),

            is_processed=True

        ).order_by('-created_at')

Phase 7  AI Integration



Add an intelligence layer on top of a system that already works. If the AI layer breaks, the rest of the system keeps running.



Model choice

Use the OpenAI API with gpt-4o-mini for cost efficiency. Don't use a local model in early implementation — network latency is predictable, local inference hardware is not.



Package

Version

Purpose

gpt-4o-mini

Summarization

Fast, cheap, good enough for report summaries

text-embedding-3-small

Similarity

1536-dimension embeddings for matching similar reports

gpt-4o

Pattern analysis

Use only for 'suggest exploit pattern' — more expensive, use sparingly



What to build

Summarization  —  For each processed report, generate a 2-3 sentence plain-English summary. Store it as HackReport.ai_summary (add migration).

Similarity matching  —  Generate embeddings for each report and store them. Compare via cosine similarity to find related incidents.

Pattern tagging  —  Ask the model to suggest exploit pattern tags based on description. Human-review before applying to the record.



Failure handling

⚠  Note:  Wrap every OpenAI API call in try/except. A network timeout or rate limit error must not crash the pipeline.

If summarization fails for a report, log the error and move on. Set ai_summary=None and retry in the next run.

Add OPENAI_API_KEY presence check at startup — fail loudly if the key is missing, before any API calls are made.

Set a per-run budget cap. If you process 500 reports at once, the cost adds up. Batch in chunks of 20-50.



Summarization pattern

def summarize_report(report):

    try:

        response = openai.chat.completions.create(

            model='gpt-4o-mini',

            messages=[{

                'role': 'system',

                'content': 'Summarize this security incident in 2-3 sentences.'

            }, {

                'role': 'user',

                'content': report.description[:2000]  # cap input length

            }],

            max_tokens=150

        )

        report.ai_summary = response.choices[0].message.content

        report.save(update_fields=['ai_summary'])

    except Exception as e:

        logger.error(f'Summarization failed for {report.id}: {e}')



Important: system must run without AI

✗  Avoid:  Never make the AI layer a required dependency. The /api/reports/ endpoint must return data whether or not OpenAI is reachable. ai_summary should be null-safe everywhere it's used.

Architecture Review



8.5 / 10

MVP Architecture Rating



The remaining 1.5 points are around maintainability and future scaling — not correctness. The core architecture is sound.



What works well

Phased build order  —  Exactly right. Database before API, API before scraper, scraper before automation.

One source first  —  Many projects fail here. Scraping five sites before proving one works reliably is a recurring mistake. This plan avoids it.

AI as optional layer  —  Placing AI at Phase 7 with an explicit requirement that the system runs without it is the correct call.

Hash-based deduplication  —  Cleaner than checking title or URL string similarity. The unique constraint at the database level makes it bulletproof.

services/ separation  —  Keeps models thin and business logic testable in isolation.



What was improved in this revision

source field  —  Added to HackReport. Filtering by provider without parsing URLs is worth the one extra field.

hash uniqueness  —  Elevated from application-level check to database-level constraint (unique=True, db_index=True).

Indexes  —  Added to title, severity, created_at, published_at, and source before the first migration.

raw_data JSONField  —  Stores source-specific metadata without schema changes per scraper.

services/ layer  —  Extracted dedup, category, and report logic out of models and views.



What to ignore until later

✗  Celery and Redis — not until Phase 5.

✗  Search optimization — not until you have 100+ real records.

✗  Multiple scrapers — not until rekt.news is reliable.

✗  AI anything — not until Phase 7.
